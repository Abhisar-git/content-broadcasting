# Content Broadcasting System (Backend)

## Implemented Features
- JWT authentication with role-based access (`principal`, `teacher`)
- Teacher content upload (JPG/PNG/GIF, max file size, required metadata)
- Approval workflow (principal approve/reject with rejection reason)
- Teacher-defined scheduling (`startTime`, `endTime`) and subject-wise rotation
- Public live broadcasting endpoint by teacher (`/api/content/live/:teacherPublicId`)
- Required edge-case handling (`No content available`)
- Pagination and filters for content listing
- Rate limiting for public live API
- Subject-wise analytics endpoint for principal
- Optional caching:
  - Redis (if `REDIS_URL` provided)
  - In-memory fallback
- Optional S3 upload support (if `STORAGE_DRIVER=s3` and AWS vars provided)

## Tech Stack
- Node.js, Express
- Sequelize ORM
- PostgreSQL / MySQL
- JWT (`jsonwebtoken`)
- Password hashing (`bcryptjs`)
- File upload (`multer`)

## Setup
1. Install dependencies:
```bash
npm install
```
2. Copy env file and configure:
```bash
cp .env.example .env
```
3. Configure database in `.env`:
   - Option A: set `DATABASE_URL` (recommended for Supabase)
   - Option B: set `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`
   - For Supabase/SSL DBs set `DB_SSL=true` (and usually `DB_SSL_REJECT_UNAUTHORIZED=false`)
4. Run app:
```bash
npm run dev
```
5. API base URL:
```text
http://localhost:4000
```

## Default Principal Account
Created automatically on first server start:
- Email: value from `DEFAULT_PRINCIPAL_EMAIL`
- Password: value from `DEFAULT_PRINCIPAL_PASSWORD`

Default values in `.env.example`:
- `principal@example.com`
- `Principal@123`

## API Summary

### Auth
- `POST /api/auth/register` (teacher register)
- `POST /api/auth/login`
- `GET /api/auth/me` (auth required)

### Teacher Content
- `POST /api/content` (teacher only, multipart `file`)
- `GET /api/content/my` (teacher only, filters + pagination)
- `GET /api/content/my/:contentId` (teacher only)

### Principal
- `GET /api/principal/content` (principal only, filters + pagination)
- `GET /api/principal/content/pending` (principal only)
- `PATCH /api/principal/content/:contentId/approve` (principal only)
- `PATCH /api/principal/content/:contentId/reject` (principal only, body: `reason`)
- `GET /api/principal/analytics/subjects` (principal only)

### Public
- `GET /api/content/live/:teacherPublicId`
- `GET /api/content/live/:teacherPublicId?subject=maths`

If no active content is found:
```json
{
  "success": true,
  "message": "No content available",
  "data": []
}
```

## Scheduling Logic
- Each teacher + subject pair has an independent slot.
- Every uploaded content gets a `rotationOrder` in that slot.
- Every item has `durationMinutes` (default from env if not provided).
- Only content that is:
  - `approved`
  - has both `startTime` and `endTime`
  - and current time is within that window  
  participates in rotation.
- Rotation loops continuously based on elapsed minutes since slot creation.
